Explicación de rebase

Resumen

Tenemos un repo tanto en local como en github

  1. En local hacemos un cambio en el archivo .html (agregamos un video)

  2. En remoto (github) también hacemos un cambio en el archivo .html (agregamos una playlist)

  3. Al momento de querer hacer git push no vamos a poder porque hay cambios en github remoto (playlist) que no lo tiene identificado el local.

  4. Entonces debemos traer los cambios de remoto al local, o en otras palabras actualizar el repo local con los cambios que se tiene en remoto (github)

  5. El comando que realiza eso es git pull, sin embargo, si solo colocamos git pull, se creará un commit más que hace referencia a que dos cambios se unieron: uno local (video) y remoto github (playlist). Ese commit de más puede causar confusión cuando se revise el historial de commits. En este contexto, dicho commit puede considerarse innecesario (no siempre), ya que el merge no aporta información clara sobre los cambios realizados, sino solo sobre la sincronización entre local y remoto.

  6. Sin embargo, si utilizamos git pull --rebase, no se creará ese commit con el merge, solo se justarán ambos commits (el del video y el de la playlist) uno encima de otro, de manera que ambos al final Git toma los commits remotos y luego reaplica los commits locales encima, manteniendo sus mensajes originales y dejando un historial lineal.

  7. En caso de que los cambios entren en conflicto (por ejemplo, en local agregamos un video y en remoto se eliminó ese mismo video), tanto git pull como git pull --rebase detectarán el conflicto, se detendrán y Git pedirá que el usuario lo resuelva manualmente eligiendo qué cambio prevalece o combinándolos.
